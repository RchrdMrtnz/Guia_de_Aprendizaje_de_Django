10. Arquitectura Moderna en Django

Ir más allá del tutorial básico es fundamental para crear aplicaciones mantenibles y escalables. Aquí explicamos cómo hemos reestructurado este proyecto para seguir estándares profesionales.

1. Modelo de Usuario Personalizado (Custom User)

Una de las primeras recomendaciones de la documentación de Django es usar un modelo de usuario personalizado en lugar del django.contrib.auth.models.User por defecto. Esto permite flexibilidad futura (por ejemplo, usar email para loguearse en lugar de username).

Implementación

Hemos creado una nueva aplicación users y definido un modelo CustomUser que hereda de AbstractUser.

# src/users/models.py
from django.contrib.auth.models import AbstractUser
from django.db import models

class CustomUser(AbstractUser):
    # Aquí podemos añadir campos extra como 'bio', 'fecha_nacimiento', etc.
    pass

Y configurado en los settings:

# src/curso_django/settings/base.py
AUTH_USER_MODEL = 'users.CustomUser'

¿Por qué hacer esto? Si empiezas un proyecto con el User por defecto y luego necesitas cambiarlo, la migración de la base de datos es extremadamente dolorosa. Hacerlo desde el día 1 es la mejor práctica.

2. Configuración Modular (Settings)

Tener un solo archivo settings.py se vuelve inmanejable rápidamente. Lo ideal es separar la configuración por entornos.

Estructura de Archivos

src/curso_django/settings/
├── base.py       # Configuraciones comunes (Apps, Middleware, i18n)
├── local.py      # Entorno de desarrollo (Debug=True, SQLite)
└── production.py # Entorno de producción (Debug=False, Postgres, Seguridad)

¿Cómo funciona?

El archivo base.py contiene todo lo que es común. local.py importa todo de base y sobreescribe lo necesario:

# src/curso_django/settings/local.py
from .base import *

DEBUG = True
ALLOWED_HOSTS = ["*"]

Para ejecutar el proyecto, Django necesita saber qué archivo usar. Esto se configura en manage.py y las variables de entorno:

# src/manage.py
os.environ.setdefault("DJANGO_SETTINGS_MODULE", "curso_django.settings.local")

3. Variables de Entorno

Nunca guardes secretos (API Keys, contraseñas de DB) en el código. Usamos python-decouple para leer un archivo .env que es ignorado por git.

# .env
SECRET_KEY=tu-clave-secreta-super-segura
DEBUG=True
ALLOWED_HOSTS=.localhost,127.0.0.1
DATABASE_URL=postgres://user:password@db:5432/dbname

Y en los settings:

from decouple import config

SECRET_KEY = config('SECRET_KEY')
DEBUG = config('DEBUG', default=False, cast=bool)
